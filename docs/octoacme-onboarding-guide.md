# OctoAcme — Team Member Onboarding Guide

## Purpose
Provide a structured onboarding experience for new team members joining OctoAcme projects. This guide ensures new members understand processes, roles, tools, and collaboration patterns quickly.

## When to use
- When a new team member joins a project
- When an existing team member transitions to a new role
- As a refresher for returning team members

---

## Onboarding Overview

A successful onboarding experience enables new team members to:
- Understand the project context, goals, and current status
- Know their role and how it fits with other team members
- Access the tools, repositories, and documentation they need
- Begin contributing meaningfully within the first sprint

---

## Week 1: Foundations

### Day 1-2: Welcome & Context

#### Administrative Setup
- [ ] Access granted to project repository
- [ ] Added to team communication channels (Slack/Teams)
- [ ] Calendar invites sent for recurring meetings
- [ ] Development environment setup instructions provided
- [ ] Security and compliance training scheduled (if required)

#### Project Introduction
- [ ] Review the [Project Management Overview](octoacme-project-management-overview.md)
- [ ] Read the Project One-pager (in project repo)
- [ ] Understand project goals, success metrics, and timeline
- [ ] Review current project status and recent updates

#### Meet the Team
- [ ] 1:1 scheduled with Project Manager
- [ ] Introduction to immediate team members
- [ ] Understand team structure and reporting lines
- [ ] Exchange contact information and availability

### Day 3-5: Role Orientation

#### Understand Your Role
- [ ] Review [Roles & Personas](octoacme-roles-and-personas.md)
- [ ] Understand your specific responsibilities
- [ ] Clarify goals and success metrics for your role
- [ ] Identify your primary collaboration partners

#### Process Familiarization
- [ ] Review [Execution & Tracking](octoacme-execution-and-tracking.md)
- [ ] Understand team rhythm (standups, planning, demos)
- [ ] Review project board and workflow columns
- [ ] Understand PR and code review conventions

---

## Week 2: Engagement

### Deepen Understanding

#### Documentation Deep Dive
- [ ] Review [Project Planning](octoacme-project-planning.md)
- [ ] Understand backlog structure and prioritization
- [ ] Review Definition of Done
- [ ] Familiarize with [Risks & Communication](octoacme-risks-and-communication.md)

#### Technical Context (for technical roles)
- [ ] Review codebase structure and architecture
- [ ] Understand build and deployment processes
- [ ] Review [Release & Deployment](octoacme-release-and-deployment.md)
- [ ] Shadow a code review or pair with a team member

### Begin Contributing

#### First Contributions
- [ ] Attend first daily standup
- [ ] Participate in sprint planning (if timing aligns)
- [ ] Pick up a small, well-defined task
- [ ] Complete first code review or deliverable

#### Ask Questions
- [ ] Document questions and seek answers from teammates
- [ ] Schedule follow-up 1:1s with key collaborators
- [ ] Identify any gaps in onboarding documentation

---

## Week 3-4: Integration

### Become Fully Productive

#### Full Participation
- [ ] Actively participate in all relevant ceremonies
- [ ] Take ownership of tasks independently
- [ ] Provide input on planning and estimation
- [ ] Contribute to discussions and decisions

#### Collaboration
- [ ] Review [Cross-Functional Collaboration Checklist](octoacme-cross-functional-collaboration-checklist.md)
- [ ] Establish working relationships with other roles
- [ ] Understand handoff patterns with collaborators
- [ ] Participate in at least one retrospective

### Feedback Loop

#### Onboarding Retrospective
- [ ] Share feedback on onboarding experience
- [ ] Identify areas for improvement
- [ ] Suggest documentation updates if needed
- [ ] Confirm readiness for full contribution

## Role-Specific Onboarding Supplements

### Developers

#### Additional Technical Setup
- [ ] Development environment configured and tested
- [ ] Access to CI/CD pipelines
- [ ] Familiar with coding standards and style guides
- [ ] Understand testing expectations and coverage requirements

#### Key Collaborators
- **QA:** Understand testing process and defect workflow
- **UX Designer:** Know how to access design specifications
- **Solution Architect:** Understand architectural patterns in use
- **Technical Writer:** Know documentation expectations

### Product Managers

#### Additional Context
- [ ] Review product roadmap and vision documents
- [ ] Understand stakeholder landscape
- [ ] Review user research and customer feedback sources
- [ ] Understand prioritization framework

#### Key Collaborators
- **Business Analyst:** Align on requirements workflow
- **UX Designer:** Establish design review cadence
- **Stakeholders:** Schedule introductory meetings
- **Developers:** Understand technical constraints

### Project Managers

#### Additional Context
- [ ] Review risk register and current risks
- [ ] Understand project dependencies and cross-team relationships
- [ ] Review historical status reports
- [ ] Understand escalation paths and contacts

#### Key Collaborators
- **Product Manager:** Establish weekly sync
- **Scrum Master:** Clarify ceremony responsibilities
- **Stakeholders:** Understand communication expectations
- **Solution Architect:** Understand technical dependencies

### QA Engineers

#### Additional Context
- [ ] Review test strategy and test plans
- [ ] Access to test environments
- [ ] Understand defect tracking workflow
- [ ] Review quality metrics and coverage expectations

#### Key Collaborators
- **Developers:** Establish defect triage process
- **Product Manager:** Clarify acceptance criteria questions
- **Technical Writer:** Coordinate on documentation testing

### Scrum Masters

#### Additional Context
- [ ] Review team's agile practices and maturity
- [ ] Understand team velocity and historical performance
- [ ] Review outstanding action items from retrospectives
- [ ] Understand team dynamics and challenges

#### Key Collaborators
- **Project Manager:** Clarify responsibilities and handoffs
- **Developers:** Understand blockers and impediments
- **Product Manager:** Align on backlog readiness

### UX Designers

#### Additional Context
- [ ] Review existing design system and style guides
- [ ] Understand user personas and research
- [ ] Access to design tools and repositories
- [ ] Review design review and approval process

#### Key Collaborators
- **Product Manager:** Understand feature priorities
- **Developers:** Establish design handoff process
- **Stakeholders:** Understand feedback channels

### Technical Writers

#### Additional Context
- [ ] Review documentation standards and style guide
- [ ] Understand documentation publishing workflow
- [ ] Access to documentation repositories
- [ ] Review existing product documentation

#### Key Collaborators
- **Developers:** Establish technical review process
- **Product Manager:** Understand feature context
- **QA:** Coordinate on accuracy validation

### Solution Architects

#### Additional Context
- [ ] Review existing architecture documentation
- [ ] Understand technical debt and known issues
- [ ] Review technology stack and standards
- [ ] Understand security and compliance requirements

#### Key Collaborators
- **Developers:** Establish design review process
- **Project Manager:** Understand project constraints
- **Stakeholders:** Understand business drivers

### Business Analysts

#### Additional Context
- [ ] Review business domain and context
- [ ] Understand requirements gathering workflow
- [ ] Access to stakeholder contact information
- [ ] Review existing requirements documentation

#### Key Collaborators
- **Stakeholders:** Establish interview schedules
- **Product Manager:** Align on requirements handoff
- **Developers:** Understand technical constraints

### Common Questions

**Q: Who do I ask about...?**
- **Project status/timeline:** Project Manager
- **Feature priorities/requirements:** Product Manager
- **Technical questions:** Tech Lead or Solution Architect
- **Process questions:** Scrum Master or Project Manager
- **Access issues:** Project Manager

**Q: Where do I find...?**
- **Project board:** Link in project repo README
- **Documentation:** `docs/` folder in repository
- **Meeting notes:** Team wiki or shared drive
- **Design files:** Ask UX Designer for access
